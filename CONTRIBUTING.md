# BodyBuddy Contributing Guide

이 문서는 **BodyBuddy 프로젝트 협업 규칙 및 GitHub 운영 방침**을 정리한 가이드입니다.  
모든 팀원은 아래 내용을 숙지하고 동일한 방식으로 협업을 진행합니다.

## 1. 브랜치 전략 — GitHub Flow
- `main` 브랜치는 항상 안정된 상태를 유지합니다.  
- 새로운 기능 개발 시 `feature/기능명` 형태의 브랜치를 생성합니다.
- 기능 구현이 완료되면 Pull Request (PR)를 생성하고
  팀원들의 코드 리뷰를 거친 후 `main` 브랜치에 병합합니다.
- 해당 기능 개발이 마무리되었다면, 생성했던 브랜치를 지웁니다.
- feature/fix는 코드, 문서, 버그 등 main에 병합하기 전 파일들을 수정할 브랜치입니다.

## 2. 커밋 메시지 컨벤션

> 모든 커밋 메시지는 한눈에 변경 목적을 파악할 수 있어야 합니다.

 `feat:`  새로운 기능 추가 
 `fix:`  버그 수정 
 `docs:`  문서 수정 (예: README, 주석 등) 
 `style:`  코드 스타일 변경 (세미콜론, 들여쓰기, 포맷 등) 
 `refactor:`  코드 리팩토링 (성능 개선, 구조 변경 등) 
 'update:' 코드, 파일 추가 개선

## 3. Pull Request (PR) 생성 및 리뷰 규칙

1. **브랜치명 규칙**
   - `브랜치명/작업내용` 형태로 작성합니다.  
     예: `feature/로그인-정보저장`, `fix/식단추천-API오류`
   
2. **PR 생성 시**
   - 작업 내용을 명확히 요약하고 변경 사항을 간단히 설명합니다.
   - 필요 시 스크린샷이나 실행 결과를 첨부해도 좋습니다.
   - 팀원 모두의 **리뷰 및 동의 후** `main`에 병합합니다.

3. **리더 역할**
   - 모든 PR을 검토 후 피드백합니다.  
   - 무조건 수락하지 않고 코드 품질, 일관성, 확장성을 고려하여 리뷰합니다.
   - 더 나은 방향이 있다면 적극적으로 제안하고 토론합니다.

4. **리뷰 주기**
   - 매주 **금요일**에 팀원 전체가 PR을 확인하고 리뷰합니다.
   - 불가피하다면, 금요일이 아니더라도 일주일 내 1회 이상은 리뷰합니다.

5. **리뷰 문화**
   - 더 나은 결과물을 위해 **객관적이고 직설적인 피드백**을 지향합니다.
   - 비판이 아닌 **개선 중심의 대화**로 피드백합니다.

## 4. 이슈 등록 규칙

> 모든 버그, 기능 제안, 개선 사항은 GitHub Issue로 관리합니다.
> 이슈 제목 형식: [타입] 이슈 설명

## 5. 협업 시 유의사항

- 브랜치 병합 전 반드시 `main`의 최신 버전을 pull 받아 충돌을 방지합니다.  
- 코드 리뷰 시, 기능 외에도 **코드 가독성 / 중복 제거 / 네이밍 규칙**도 함께 확인합니다.  
- 불필요한 파일(`__pycache__`, `.idea`, `.DS_Store` 등)은 `.gitignore`로 관리합니다.

## 릴리즈 관리
버전 태깅은 Semantic Versioning(v1.0.0)을 따릅니다.

## 문의 및 토론
프로젝트 관련 제안이나 논의는 **GitHub Discussions** 또는 **Notion 코멘트**를 통해 진행합니다.  
협업 중 발생한 문제는 팀원 간 빠른 커뮤니케이션으로 해결합니다.
